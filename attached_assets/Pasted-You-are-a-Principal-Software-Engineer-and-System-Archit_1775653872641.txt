## 🧩 TECHNICAL SPECIFICATIONS (Derived from Deep Analysis - ALL PHASES)

### PHASE 1: REQUIREMENT BREAKDOWN
- **Core Modules**: Auth, Subscription (Stripe), Score Management (only last 5 scores, 1-45 Stableford), Draw Engine (Random + Weighted by user score frequency), Prize Pool (40/35/25 split, jackpot rollover for 5-match), Charity System (min 10% contribution, user selectable), User Dashboard, Admin Panel, Winner Verification.
- **Hidden Requirements**: Concurrency handling in draw (Redis lock), Fraud detection (score validation, admin review), Payment webhook security (signature verification, idempotency), Timezone-based monthly draw execution (cron job), Data consistency (score rolling logic).
- **Edge Cases**: Duplicate webhooks, <5 scores, no winners → rollover, multiple winners split, expired subscription blocking, duplicate score handling allowed but validated.

### PHASE 2: SYSTEM ARCHITECTURE
- **Modular Monolith** (easy split to microservices later).
- **Frontend**: React + Vite + Tailwind CSS + Redux Toolkit (RTK Query) + Framer Motion.
- **Backend**: Node.js + Express, MongoDB + Mongoose, Redis (for draw lock & queue), JWT auth, Stripe SDK, Nodemailer, Multer + Cloudinary for uploads.
- **Folder Structure**:
  - `backend/`: controllers, services, models, routes, middleware, utils, config
  - `frontend/`: src/components, pages, redux, utils, hooks, assets

### PHASE 3: DATABASE DESIGN (Mongoose Schemas)
Generate exact schemas for:
- **User**: name, email, password (hashed), role (USER/ADMIN), charityId (ref), contributionPercentage (min10), isActive, timestamps.
- **Subscription**: userId (unique), stripeCustomerId, stripeSubscriptionId, plan (MONTHLY/YEARLY), status, startDate, endDate, cancelAtPeriodEnd, amount, timestamps.
- **Score**: userId, score (1-45), date (not future), createdAt. Index: {userId, date:-1}. Pre-save hook to keep only last 5 scores.
- **Draw**: month, year, drawNumbers (5 unique 1-45), type (RANDOM/WEIGHTED), status (PENDING/PUBLISHED), prizePool, rolloverAmount, publishedAt. Unique index on month+year.
- **Winning**: userId, drawId, matchCount (3/4/5), amount, status (PENDING/APPROVED/REJECTED/PAID), proofImage (URL), paidAt.
- **Charity**: name, description, image (URL), website, isActive, totalDonated, featured.

### PHASE 4: AUTH & SECURITY
- JWT stored in HTTP-only cookie. Middleware for auth, role-based access (USER, ADMIN), subscription status check on protected routes.
- Rate limiting (express-rate-limit), helmet, Joi validation, bcrypt.

### PHASE 5: PAYMENT & SUBSCRIPTION (Stripe)

- Subscription lifecycle: active/inactive/cancelled/expired. Real-time status check middleware.
